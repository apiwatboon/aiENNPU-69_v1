เทรนจริงให้เป็น: Overfitting และการวัดผลโมเดล

โมเดลที่ "จำข้อสอบ" vs โมเดลที่ "เข้าใจบทเรียน"

🎯 เป้าหมายสัปดาห์นี้ — เรียนจบแล้วนักศึกษาต้อง:
  1. อธิบาย overfitting ด้วยตัวอย่างของตัวเองได้ และดูกราฟ train/test loss แล้วชี้จุดที่เริ่ม overfit ได้
  2. บอกได้ว่าทำไมต้องแยกข้อมูล train/test และห้ามให้โมเดล "เห็นข้อสอบ" เด็ดขาด
  3. เทรนโมเดลแรกของตัวเองใน Google Colab สำเร็จ

ส่วนที่ 1 — แนวคิด แนวคิด

คำถามเปิดคาบ: "เพื่อนสองคน คนแรกท่องเฉลยข้อสอบเก่าทุกข้อแบบจำเป๊ะ คนที่สองทำความเข้าใจหลักการ — ถ้าข้อสอบปีนี้เปลี่ยนโจทย์ ใครรอด?"

Overfitting = จำข้อสอบ ไม่ได้เข้าใจบทเรียน

โมเดลที่ซับซ้อนมาก (weight เยอะ) สามารถ "จำ" ข้อมูลฝึกได้หมดเปลือก รวมถึง noise และความบังเอิญ — loss บนข้อมูลฝึกต่ำสวยงาม แต่พอเจอข้อมูลใหม่กลับพังยับ นี่คือ overfitting

  • Train set — ข้อมูลให้โมเดลเรียน (แบบฝึกหัด)
  • Test set — ข้อมูลที่ซ่อนไว้วัดผล (ข้อสอบจริง) — โมเดลห้ามเห็นตอนเทรนเด็ดขาด
  • สัญญาณ overfit: train loss ลงต่อเรื่อยๆ แต่ test loss เริ่มเงยหัวขึ้น

วิธีสู้: เพิ่มข้อมูล, ลดขนาดโมเดล, regularization, dropout, early stopping — แก่นเดียวกันหมดคือ "บังคับให้เรียนหลักการ อย่าให้จำ"

จุดเชื่อมสู่ LLM: นี่คือคำตอบว่าทำไม LLM ต้องเทรนด้วยข้อความ "ระดับทั้งอินเทอร์เน็ต" — โมเดลแสนล้าน parameter จำข้อมูลเล็กๆ ได้สบาย ทางเดียวที่บังคับให้มัน "เข้าใจภาษา" คือป้อนข้อมูลมากจนจำไม่ไหว ต้องหากฎเอาเอง และปรากฏการณ์ LLM ท่องข้อมูลส่วนตัว/ลิขสิทธิ์ออกมาได้ ก็คือ overfitting บางส่วนนั่นเอง

👣 เข้าใจทีละขั้น — สอนเครื่องไม่ให้จำข้อสอบ ใน 6 ขั้น

ขั้นตอนนี้คือสิ่งที่วิศวกร ML ทำจริงทุกโปรเจกต์ — จำลำดับนี้ได้ = ทำงานเป็น

  1. แบ่งข้อมูลก่อนทำอะไรทั้งสิ้น มีข้อมูล 100 แถว → สุ่มแบ่งเป็น train 80 แถว (ให้โมเดลเรียน) กับ test 20 แถว (ซ่อนไว้ในลิ้นชัก)
    กฎเหล็ก: โมเดลห้ามเห็น test เด็ดขาด — เหมือนข้อสอบจริงที่ห้ามหลุดก่อนสอบ
  2. เทรนด้วย train เท่านั้น ใช้ความรู้ที่ผ่านมา (ตั้งแต่ Regression จนถึง Backpropagation): ปรับ weight จน train loss ลดลงเรื่อยๆ — ตรงนี้โมเดลจะดูเก่งขึ้นเสมอ (เหมือนนักเรียนที่ทำแบบฝึกหัดชุดเดิมซ้ำๆ คะแนนชุดนั้นย่อมดีขึ้น)
  3. เปิดลิ้นชัก: วัดด้วย test เอาโมเดลไปทำนายข้อมูล 20 แถวที่มันไม่เคยเห็น แล้ววัด loss — ตัวเลขนี้คือ "ความเข้าใจจริง"
  4. อ่านอาการจากตัวเลขคู่เดียว
    train losstest lossวินิจฉัย
    0.020.95😵 overfit — จำข้อสอบ (เก่งเฉพาะที่เคยเห็น)
    0.400.42✅ พอดี — เข้าใจหลักการ
    0.900.91😴 underfit — โมเดลง่ายเกิน/เทรนน้อยเกิน
    วินิจฉัยเอง: train 0.1 / test 0.6 เป็นอาการอะไร ควรทำอะไรต่อ?
  5. แก้ตามอาการ overfit → เพิ่มข้อมูล / ลดขนาดโมเดล / dropout / หยุดเทรนตรงจุดที่ test loss เริ่มเงย (early stopping)
    underfit → ขยายโมเดล / เทรนนานขึ้น — สังเกตว่าวิธีแก้สองโรคนี้ตรงข้ามกัน วินิจฉัยผิด = แก้ผิดทาง
  6. เห็นด้วยตาตัวเอง demo ข้างล่าง: จุดเขียว = train, จุดส้ม = test — เลื่อนดีกรีต่ำ (เส้นทื่อ พลาดทั้งคู่ = underfit) ไปดีกรีสูง (เส้นบิดไล่เก็บจุดเขียวทุกจุด แต่พังกับจุดส้ม = overfit)
    กด ▶ ไล่ดีกรีอัตโนมัติ แล้วจดว่า "ดีกรีไหน test error ต่ำสุด" — นั่นคือจุดพอดีของข้อมูลชุดนี้

🧩 ตัวอย่างหลายแบบ — ฝึกวินิจฉัยโมเดลเหมือนหมอ

ทุกข้อ: วินิจฉัยก่อน (overfit / underfit / อื่นๆ) แล้วสั่งยา — ค่อยกางเฉลย

ตัวอย่าง 1 — แอปสแกนใบหน้าเข้างาน

เทรนกับรูปพนักงานถ่ายในออฟฟิศตอนกลางวัน ได้ accuracy 99% (test set ก็รูปออฟฟิศกลางวัน) พอใช้จริงตอนกะดึก สแกนไม่ผ่านเพียบ — เกิดอะไรขึ้น?

ขั้น 1 — เช็คคู่ train/test ก่อน: 99% ทั้งคู่ ช่องว่างแคบ → ไม่ใช่ overfit แบบคลาสสิก
ขั้น 2 — เทียบ "โลกของข้อมูล" กับ "โลกของงานจริง": ข้อมูลทั้งหมด = กลางวัน | งานจริง = มีกลางคืนด้วย — test ไม่ได้เป็นตัวแทนงานจริง
ขั้น 3 — ตั้งชื่อโรค: distribution shift — ญาติสนิทของ overfitting: โมเดลเก่งเฉพาะสภาพแวดล้อมที่เคยเห็น
✅ คำตอบ: ตัวเลข 99% ไม่ได้โกหก แต่มันวัดผิดโลก — ยา: เก็บข้อมูลครอบคลุมสภาพใช้จริง (รวมกะดึก) แล้วแบ่ง test ให้สะท้อนงานจริง
ตัวอย่าง 2 — โมเดลทำนายราคาหุ้น train loss 0.05 / test loss 0.90

นักศึกษาคนหนึ่งภูมิใจว่า train loss ต่ำมาก — วินิจฉัย แล้วตอบเพิ่ม: ถ้าเพิ่มข้อมูลอีก 10 เท่า จะหายไหม?

ขั้น 1 — วินิจฉัยตามตำรา: ช่องว่าง 0.05 ต่อ 0.90 กว้างมาก → overfit ชัดเจน (จำแพตเทิร์นเฉพาะของข้อมูลฝึก)
ขั้น 2 — ตอบคำถามชั้นสอง: เพิ่มข้อมูล 10 เท่าหายไหม? ช่องว่างจะแคบลงจริง แต่ test loss อาจยังสูงอยู่ดี
ขั้น 3 — หาเหตุผล: ราคาหุ้นมี noise มหาศาลและแพตเทิร์นเปลี่ยนตลอด — ส่วนที่ "ทำนายได้จริง" มีน้อยมากโดยธรรมชาติ ข้อมูลเพิ่มช่วยลดการจำ แต่สร้างสัญญาณที่ไม่มีอยู่จริงไม่ได้
✅ คำตอบ: overfit + โจทย์ที่เพดานต่ำ — บางโจทย์ขีดจำกัดอยู่ที่ธรรมชาติข้อมูล ไม่ใช่โมเดล วิศวกรที่ดีรู้จักพูดว่า "โจทย์นี้ทำนายไม่ได้" ก่อนเผาเงินเพิ่ม
ตัวอย่าง 3 — เพื่อนสองคนเตรียมสอบ (overfitting ฉบับมนุษย์)

คนแรกท่องเฉลยข้อสอบเก่า 100 ข้อจนจำได้ทุกตัวอักษร คนที่สองทำข้อสอบเก่าแค่ 30 ข้อแต่สรุปหลักการเอง — ใครคือ overfit? แล้ว "ข้อสอบเก่าหลุดมาตรงๆ 1 ข้อ" ใครได้เปรียบ?

ขั้น 1 — เทียบนิยาม: คนแรก = train accuracy 100% (จำเฉลยเป๊ะ) แต่โจทย์พลิกนิดเดียวพัง → overfit | คนที่สอง = ทำข้อมูลฝึกได้ไม่เต็มแต่หลักการแน่น → generalize
ขั้น 2 — กรณีข้อสอบหลุดตรงๆ: คนแรกชนะข้อนั้นทันที — ความจำเป๊ะเอาชนะความเข้าใจ เมื่อข้อสอบ = แบบฝึกหัด
ขั้น 3 — โยงเข้า ML: "ข้อสอบหลุด" = data leakage — โมเดล overfit ทำคะแนนสวยกับข้อมูลที่รั่วจาก train หลอกให้เราคิดว่ามันเก่ง
✅ คำตอบ: คนแรก overfit และได้เปรียบเฉพาะเมื่อข้อสอบรั่ว — นี่แหละเหตุผลที่ต้องแยก test ให้สะอาดจริงๆ เพราะคะแนนจากข้อสอบรั่วไม่ได้วัดอะไรเลย
ตัวอย่าง 4 — ทำไมต้องมีชุดที่สาม (validation set)

เราลองโมเดล 20 แบบ แล้วเลือกตัวที่ทำคะแนน test ดีสุด — ฟังดูถูกต้อง แต่มีกับดักอะไรซ่อนอยู่?

ขั้น 1 — มองการเลือกเป็นการเรียนรู้: ทุกครั้งที่ดูคะแนน test แล้วตัดสินใจ เราป้อนข้อมูลจาก test กลับเข้ากระบวนการ — 20 ครั้ง = เรียนจาก test ไป 20 รอบโดยไม่รู้ตัว
ขั้น 2 — ดูผลเสีย: ตัวที่ชนะอาจแค่ "ฟลุกพอดีกับ test ชุดนี้" — คะแนนที่รายงานจะสวยเกินจริงเสมอ
ขั้น 3 — วิธีถูก: แบ่งสามชุด train / validation / test (70/15/15) — validation ใช้เลือกโมเดลกี่รอบก็ได้ ส่วน test เปิดซองครั้งเดียวตอนจบเพื่อรายงานผล
✅ คำตอบ: กับดักคือ "เลือกตาม test = แอบเรียนจาก test" — test คือข้อสอบจริงที่เปิดซองได้หนเดียว
ตัวอย่าง 5 — อ่านกราฟแล้วสั่งหยุด (early stopping)

เทรน 50 epoch: val loss ลดมาตลอด ต่ำสุดที่ epoch 17 แล้วค่อยๆ เงยขึ้น — ควรใช้โมเดลของ epoch ไหน?

ขั้น 1 — หาจุดเปลี่ยนบนกราฟ: val loss ลงเรื่อยๆ จนต่ำสุดที่ epoch 17 แล้วเริ่มเงย — จุดเงยคือสัญญาณ "เริ่มจำข้อมูลฝึก"
ขั้น 2 — ตีความสองช่วง: ก่อน 17: เรียนหลักการ (ดีขึ้นกับข้อมูลใหม่ด้วย) | หลัง 17: จำรายละเอียดเฉพาะ (train ลดต่อ แต่ val แย่ลง = ช่องว่างถ่าง)
ขั้น 3 — เลือกโมเดล: ใช้เวอร์ชันที่เซฟไว้ตอน epoch 17 ไม่ใช่ตัวสุดท้าย
✅ คำตอบ: epoch 17 — เทคนิคจริง: เซฟทุกครั้งที่ val ทำสถิติใหม่ + หยุดเมื่อไม่ดีขึ้นติดต่อกัน n รอบ (patience) กันโมเดลเสียฟรีแถมประหยัดค่าเครื่อง — ลองใส่ใน Colab การบ้านข้อ 3 ได้
ตัวอย่าง 6 — dropout: ติวกลุ่มที่สมาชิกโดดเรียนสุ่มๆ

dropout ปิดนิวรอนแบบสุ่ม 30% ทุกๆ รอบเทรน — ทำไมการ "ทำลายเครือข่ายชั่วคราว" ถึงช่วยลด overfit?

ขั้น 1 — ดูปัญหาที่เกิดถ้าไม่มี dropout: นิวรอนบางตัว "รับจำ" ลายเฉพาะของข้อมูลฝึก ตัวอื่นพึ่งมันแล้วว่างงาน — ความรู้กระจุก เปราะบาง
ขั้น 2 — ใส่ dropout: ทุกๆ รอบเทรน สุ่มปิดนิวรอน 30% — "ตัวรับจำ" หายไปบ้าง นิวรอนอื่นถูกบังคับให้ทำงานแทน
ขั้น 3 — ผลระยะยาว: ไม่มีตัวไหนพึ่งตัวไหนได้ → ความรู้ถูกเกลี่ยทั่วเครือข่าย → เจอข้อมูลใหม่แล้วทนทานกว่า
✅ คำตอบ: เหมือนติวกลุ่มที่สมาชิกโดดเรียนไม่ซ้ำหน้าทุกนัด — ไม่มีใครพึ่ง "เพื่อนคนเก่งคนเดียว" ได้ ทุกคนเลยต้องเข้าใจเองหมด = เครือข่ายที่ generalize
ตัวอย่าง 7 — เสกข้อมูลเพิ่มฟรี (data augmentation) และจุดที่ห้ามใช้

รูปแมว 1,000 ใบ → พลิกซ้ายขวา หมุนเล็กน้อย ครอป ปรับแสง = หลายหมื่นใบ — แล้วทำแบบเดียวกันกับรูป "ป้ายจราจร" ได้ไหม?

ขั้น 1 — ตั้งคำถามทดสอบ: "หลังแปลงภาพแล้ว ป้ายคำตอบ (label) ยังถูกอยู่ไหม?"
ขั้น 2 — เคสแมว: พลิกซ้ายขวา → ยังเป็นแมว ✓ หมุนนิด ครอป ปรับแสง → ยังเป็นแมว ✓ ใช้ได้หมด
ขั้น 3 — เคสป้ายจราจร: ป้าย "เลี้ยวซ้าย" พลิกซ้ายขวา → กลายเป็นป้าย "เลี้ยวขวา" แต่ label ยังเขียนว่า "เลี้ยวซ้าย" — สอนโมเดลผิด! ✗
✅ คำตอบ: แมวได้ ป้ายจราจรไม่ได้ (สำหรับการพลิก) — กฎ: augmentation ต้องเป็นการแปลงที่ "ความหมายไม่เปลี่ยน" ในโจทย์นั้นๆ เครื่องมือดีแค่ไหนก็ต้องคิดก่อนใช้
ตัวอย่าง 8 — ข้อมูลน้อยใช้ k-fold

มีข้อมูลแค่ 100 แถว แบ่ง test 20 แถวก็เสียดาย ข้อมูลฝึกเหลือนิดเดียว — มีวิธีใช้ข้อมูลให้คุ้มกว่านี้ไหม?

ขั้น 1 — แบ่งข้อมูล 100 แถวเป็น 5 ก้อน: ก้อนละ 20 แถว (A, B, C, D, E)
ขั้น 2 — ผลัดกันเป็นข้อสอบ: รอบ 1: เทรนด้วย B–E สอบด้วย A | รอบ 2: เทรน A,C–E สอบ B | ... ครบ 5 รอบ
ขั้น 3 — สรุปคะแนน: เฉลี่ยคะแนนสอบทั้ง 5 รอบ — ทุกแถวได้เป็นทั้งข้อมูลฝึกและข้อสอบ (คนละรอบ ไม่เคยซ้อนกัน)
✅ คำตอบ: 5-fold cross-validation — ผลวัดน่าเชื่อถือขึ้นมากสำหรับข้อมูลเล็ก แลกกับเวลาเทรน ×5 มาตรฐานของงานวิจัยที่ข้อมูลหายาก เช่น การแพทย์
ตัวอย่าง 9 — วินิจฉัยข้อยาก: train 55% / test 54% (เดาสุ่มได้ 50%)

ไม่ overfit แน่ (ช่องว่างแคบนิดเดียว) — แต่โมเดลนี้โอเคหรือยัง? เป็นอาการอะไรได้บ้าง?

ขั้น 1 — ตัดข้อสงสัยแรก: ช่องว่าง train/test แค่ 1% → ไม่ใช่ overfit แน่นอน
ขั้น 2 — เทียบกับเส้นฐาน: เดาสุ่มได้ 50% โมเดลได้ 54–55% — ดีกว่าสุ่มแค่นิดเดียว = แทบไม่เรียนรู้อะไร
ขั้น 3 — แยกสองสาเหตุที่เป็นไปได้: (1) underfit: โมเดลง่าย/เทรนน้อยเกิน (2) feature ไม่มีสัญญาณ: ต่อให้โมเดลเทพก็ทำนายไม่ได้ (เช่น ทำนายหวยจากวันเกิด)
ขั้น 4 — วิธีทดสอบแยกโรค: ขยายโมเดล/เทรนนานขึ้น แล้วดู train accuracy — ถ้ายังไม่ขึ้น แสดงว่าเป็นข้อ (2)
✅ คำตอบ: underfit หรือ feature ไร้สัญญาณ — ถ้าเป็นอย่างหลัง ต้องเปลี่ยน feature ไม่ใช่เปลี่ยนโมเดล
ตัวอย่าง 10 — ข้อสอบรั่วระดับวงการ: LLM กับ benchmark

LLM ถูกอวดว่าทำข้อสอบมาตรฐาน (benchmark) ได้คะแนนสูงมาก — แต่ข้อสอบพวกนั้นโพสต์อยู่ในอินเทอร์เน็ตที่โมเดลใช้เทรน... คุ้นๆ ไหม?

ขั้น 1 — ไล่เส้นทางข้อมูล: ข้อสอบ benchmark ถูกโพสต์ในเว็บ/วารสาร → เว็บถูกกวาดมาเป็นข้อมูล pretrain → ข้อสอบ (และเฉลย) อยู่ในข้อมูลฝึกแล้ว
ขั้น 2 — เทียบกับหลักสัปดาห์นี้: test ที่โมเดลเคยเห็นตอนเทรน = data leakage ตรงตามนิยาม — แค่สเกลระดับอินเทอร์เน็ต (เรียกว่า benchmark contamination)
ขั้น 3 — ผลที่ตามมา: คะแนนสะท้อน "ความจำ" ไม่ใช่ "ความสามารถ" — เทียบระหว่างโมเดลก็ไม่แฟร์ (ใครกวาดเว็บใหม่กว่าได้เปรียบ)
✅ คำตอบ: วงการแก้ด้วยข้อสอบลับ/ข้อสอบสร้างใหม่สดๆ — บทเรียนติดตัว: เห็นตัวเลข benchmark ให้ถามเสมอว่า "test สะอาดแค่ไหน" หลักการเดียวกับที่คุณใช้ตรวจ notebook เพื่อน

ส่วนที่ 2 — ทดลองเล่น ทดลองเล่น

🧪 เลือกความซับซ้อนของโมเดล: พอดี หรือ จำข้อสอบ?

จุดเขียว = ข้อมูลฝึก, จุดส้ม = ข้อมูลทดสอบ (โมเดลไม่เคยเห็น) — เลื่อน "ดีกรีของโมเดล" (ความซับซ้อน) แล้วเฝ้าดู error สองค่าแยกทางกัน

ความซับซ้อนของโมเดล (ดีกรีพหุนาม):

🎬 ห้องแล็บเทรนสด — ดู overfitting เกิดต่อหน้าต่อตา

demo ข้างบนเลื่อน "ความซับซ้อน" ด้วยมือ — คราวนี้มาดูของจริงกว่านั้น: ระหว่างเทรน ความซับซ้อนจะค่อยๆ เพิ่มเอง กด ▶ เทรน แล้วดูเส้น loss สองเส้นถูกวาดสดทีละ epoch: เขียว = train loss (ลดลงตลอด เพราะโมเดลจำเก่งขึ้นเรื่อยๆ) · ส้ม = validation loss (ความเข้าใจจริง — ลดแล้วเงยหัวขึ้นเมื่อโมเดลเริ่มจำ noise)
โปร่งใสเรื่องวิธีจำลอง: ทุก epoch เครื่อง fit โมเดลพหุนามกับข้อมูลจริงแล้ววัด loss จริงทั้งสองค่า โดยจำลอง "การเทรนนานขึ้น" ด้วยการค่อยๆ คลาย regularization (เบรกกันจำ) ลง — พฤติกรรมเดียวกับ gradient descent ที่ยิ่งเทรนนานโมเดลยิ่งซับซ้อนขึ้น

👆 โมเดลตอนนี้ (เส้นน้ำเงิน) ทาบกับข้อมูล — พอเริ่ม overfit เส้นจะ "บิดตัวไล่เก็บ" จุดเขียว (train) ทีละจุด แล้วยิ่งพลาดจุดส้ม (val)

ลอง: เทรนโมเดล "ใหญ่" จนจบ — พอเส้นส้มเงยขึ้นถาวร จะมีเส้นประ ✂️ ปรากฏ = จุดที่ early stopping ควรสั่งหยุด แต่เราปล่อยเทรนต่อให้เห็นช่องว่างถ่างออกเรื่อยๆ · แล้วรีเซ็ตลอง "เล็ก" — ทำไมมันไม่ overfit เลย (แต่ loss ก็ตันอยู่สูง = underfit)? · ปิดท้ายด้วย "กลาง" แล้วเทียบว่า val ต่ำสุด ของขนาดไหนดีที่สุดสำหรับข้อมูลชุดนี้

✂️ แบ่งข้อมูลเอง — train / validation / test

ข้อมูล 40 จุด สองคลาส: คลาส A (24 จุด) · คลาส B (16 จุด) — สีขอบของแต่ละจุดบอกว่ามันถูกหั่นไปกองไหน: train / validation / test · เลื่อนสไลเดอร์สองตัวเพื่อย้ายเส้นแบ่งเอง แล้วอ่านคำวินิจฉัยข้างล่าง

ตอนนี้ข้อมูล "เรียงตามคลาส" อยู่ — ดูคำเตือนสิ: test กลายเป็นคลาส B ล้วน! กด 🎲 สุ่มสับ แล้วดูสัดส่วนคลาสในทุกกองกลับมาใกล้ 60/40 · จากนั้นลองลากให้ test เหลือน้อยกว่า 10% แล้วอ่านคำเตือน — ข้อสอบแค่ 2–3 ข้อวัดอะไรไม่ได้จริง

🎲 Dropout — ติวกลุ่มที่สมาชิกโดดเรียน

จากตัวอย่างที่ 6 ข้างบน: dropout คือการสุ่มปิดนิวรอนชั้นซ่อนทุกรอบเทรน — กด ▶ แล้วดูว่าแต่ละรอบ (0.7 วินาที = 1 รอบเทรน) ใคร "โดดเรียน" บ้าง เส้นที่ต่อกับคนโดดหายไปด้วย ที่เหลือถูกบังคับให้ทำงานแทนกัน — ไม่มีใครได้พึ่ง "เพื่อนคนเก่งคนเดียว"

อัตราโดดเรียน: 30%

เครือข่าย 4-6-6-2 พร้อมแล้ว — ตอนนี้ทุกนิวรอนมาเรียนครบ

ลอง: ตั้งอัตรา 0% (= ไม่ใช้ dropout) เทียบกับ 50% — ที่ 50% สังเกตว่าเครือข่ายที่เหลือแต่ละรอบ "หน้าตาไม่ซ้ำกันเลย" เท่ากับเราเทรนเครือข่ายย่อยคนละตัวนับพันแบบแล้วเฉลี่ยรวมกัน · ตอนใช้งานจริง (inference) ไม่มีใครโดด แต่ต้องลดกำลังทุกคนลงตามอัตราที่เคยโดด

🏔️ ภูมิประเทศ loss แบบ 3D — หุบแคบ "จำ" หุบกว้าง "เข้าใจ"

พื้นผิวนี้คือ loss ของ weight ทุกค่าที่เป็นไปได้ (ยิ่งต่ำยิ่งดี) — มีหุบแคบแต่ลึก (weight ที่จำข้อมูลเทรนได้เป๊ะ loss ต่ำสุดๆ) กับหุบกว้าง (weight ที่จับหลักการ) · ลากเมาส์/นิ้วบนภาพเพื่อหมุนดู · ปล่อยลูกบอล = เทรนจนไหลลงก้นหุบ · แล้วกด 🌊 สลับข้อมูล: ข้อมูลใหม่ทำให้ภูมิประเทศขยับนิดเดียว — ลูกบอล (weight) อยู่ที่เดิม แต่ loss จะกระโดดแค่ไหน ขึ้นกับว่าก้นหุบที่เลือกไว้กว้างหรือแคบ

ยังไม่มีลูกบอล — กดปล่อยบอลสักลูก (หรือทั้งสองลูก จะได้เทียบกันชัดๆ)
สูตรชมช็อต "โอ้โห": ปล่อยบอลทั้งสองลูก รอให้นิ่งก้นหุบ → กด 🌊 — บอลแดง (หุบแคบ) loss พุ่งขึ้นหลายเท่าเพราะภูเขาขยับแล้วมันไปค้างบนผนังหุบ ส่วนบอลน้ำเงิน (หุบกว้าง) แทบไม่สะเทือน · นี่คือภาพเบื้องหลังว่าทำไมเทคนิคที่ดันโมเดลไปหาหุบกว้าง (dropout, ข้อมูลเยอะ, โมเดลไม่ใหญ่เกิน) ถึง generalize ดีกว่า

ส่วนที่ 3 — โค้ดจริง: เอาไปรันใน Colab ได้เลย ลงมือจริง

💻 สคริปต์เดียวจบ — โมเดล "จำข้อสอบ" ปะทะ โมเดลใส่เกราะ Dropout + EarlyStopping

คัดลอกทั้งก้อนไปวางใน Google Colab แล้วกด Run ได้ทันที (ไม่ต้องแก้อะไรสักบรรทัด ใช้เวลา ~2–3 นาที) — นี่คือ "ห้องแล็บเทรนสด" เวอร์ชันของจริงบนข้อมูลลายมือ MNIST

# ============================================================
# ดู overfitting ของจริง แล้วปราบด้วย Dropout + EarlyStopping
# วางทั้งก้อนใน Google Colab แล้วกด Run ได้เลย (~2-3 นาที)
# ============================================================
import matplotlib.pyplot as plt
from tensorflow import keras
from tensorflow.keras import layers

# 1) โหลดข้อมูลลายมือเลข MNIST แล้วสเกลพิกเซลเป็นช่วง 0-1
(x_train, y_train), (x_test, y_test) = keras.datasets.mnist.load_data()
x_train = x_train.reshape(-1, 784).astype("float32") / 255.0
x_test  = x_test.reshape(-1, 784).astype("float32") / 255.0

# ตั้งใจใช้แค่ 5,000 ภาพ — ข้อมูลน้อย + โมเดลใหญ่ = overfit ให้เห็นชัดๆ
x_small, y_small = x_train[:5000], y_train[:5000]

def build_model(use_dropout=False):
    m = keras.Sequential([layers.Input(shape=(784,)),
                          layers.Dense(512, activation="relu")])
    if use_dropout:
        m.add(layers.Dropout(0.5))   # ติวกลุ่มที่สมาชิกโดดเรียน 50%
    m.add(layers.Dense(512, activation="relu"))
    if use_dropout:
        m.add(layers.Dropout(0.5))
    m.add(layers.Dense(10, activation="softmax"))
    m.compile(optimizer="adam", loss="sparse_categorical_crossentropy",
              metrics=["accuracy"])
    return m

# 2) รุ่น A: โมเดลใหญ่ ไม่มีเกราะกันจำ — ปล่อยเทรนยาว 30 epoch
model_a = build_model(use_dropout=False)
hist_a = model_a.fit(x_small, y_small, epochs=30, batch_size=128,
                     validation_split=0.2, verbose=0)

# 3) รุ่น B: โมเดลเดียวกัน + Dropout + EarlyStopping (val ไม่ดีขึ้น 3 รอบติด = หยุด)
early = keras.callbacks.EarlyStopping(monitor="val_loss", patience=3,
                                      restore_best_weights=True)
model_b = build_model(use_dropout=True)
hist_b = model_b.fit(x_small, y_small, epochs=30, batch_size=128,
                     validation_split=0.2, callbacks=[early], verbose=0)

# 4) พล็อตเทียบ — ภาพเดียวกับ "ห้องแล็บเทรนสด" ในเว็บสัปดาห์นี้
plt.figure(figsize=(10, 5))
plt.plot(hist_a.history["loss"], "g--", label="A train")
plt.plot(hist_a.history["val_loss"], "r-", lw=2.5, label="A val (overfit!)")
plt.plot(hist_b.history["loss"], "c--", label="B train")
plt.plot(hist_b.history["val_loss"], "b-", lw=2.5, label="B val (dropout + early stop)")
plt.xlabel("epoch"); plt.ylabel("loss"); plt.legend(); plt.grid(alpha=0.3)
plt.title("A: memorize  vs  B: dropout + early stopping")
plt.show()

# 5) เปิดซอง test ครั้งเดียวตอนจบ เพื่อรายงานผลจริง
print("Test accuracy A:", round(model_a.evaluate(x_test, y_test, verbose=0)[1], 4))
print("Test accuracy B:", round(model_b.evaluate(x_test, y_test, verbose=0)[1], 4))

สิ่งที่ต้องสังเกตตอนรัน:

  • กราฟรุ่น A: เส้น train (เขียวประ) ดิ่งเข้าใกล้ศูนย์ แต่ val (แดง) เงยหัวขึ้นหลังไม่กี่ epoch — ภาพเดียวกับ "ห้องแล็บเทรนสด" ข้างบนเป๊ะ ลองชี้จุดที่ควร early stop บนกราฟจริงดู
  • รุ่น B เส้นสั้นกว่า เพราะ EarlyStopping สั่งหยุดเองก่อนครบ 30 epoch (val ไม่ดีขึ้น 3 รอบติด) — แถม val loss ของ B ต่ำกว่า A ทั้งที่ "เทรนน้อยกว่า"
  • สองบรรทัดสุดท้าย: test accuracy ของ B ควรชนะ A — และสังเกตว่าเราแตะ test แค่ครั้งเดียวตอนจบเพื่อรายงานผล ไม่เอามาเลือกโมเดล (หลักการจากตัวอย่างที่ 4)

ส่วนที่ 4 — แบบฝึกหัด แบบฝึกหัด

📝 ข้อ 1 (ทำในคาบ) — อ่านกราฟวินิจฉัยโมเดล

โมเดล A: train loss 0.02, test loss 0.95 | โมเดล B: train loss 0.40, test loss 0.42 | โมเดล C: train loss 0.90, test loss 0.91
จงวินิจฉัยแต่ละโมเดล (overfit / พอดี / underfit) และเสนอวิธีแก้ของ A กับ C

ดูเฉลย A: overfit (train ดีเลิศ test แย่) → เพิ่มข้อมูล / ลดขนาดโมเดล / regularization
B: พอดี (สองค่าใกล้กันและต่ำ)
C: underfit (แย่ทั้งคู่ — โมเดลง่ายเกินหรือเทรนน้อยเกิน) → เพิ่มความซับซ้อน / เทรนนานขึ้น
📝 ข้อ 2 (ทำในคาบ) — จับผิดการแบ่งข้อมูล

นักวิจัยคนหนึ่งสร้างโมเดลทำนายราคาหุ้น เทรนด้วยข้อมูลปี 2020–2024 แล้วทดสอบด้วยข้อมูล...ปี 2022 ได้ accuracy 99% จึงประกาศความสำเร็จ — มีอะไรผิด และผลจริงจะเป็นอย่างไรเมื่อใช้งาน?

ตอบคำถามใหญ่ได้แล้วว่า "ทำไม LLM ต้องกินข้อมูลทั้งอินเทอร์เน็ต" — เพราะโมเดลยิ่งใหญ่ยิ่งจำเก่ง ต้องป้อนข้อมูลมากกว่าที่มันจำไหว มันถึงจะยอม "เข้าใจ" และเราเทรนโมเดลจริงเป็นแล้วด้วย Colab
สัปดาห์หน้า: MLP มองภาพ 28×28 เป็นตัวเลขเรียงแถว 784 ช่อง — มันไม่รู้เลยว่า pixel ข้างกันเกี่ยวข้องกัน เราต้องการสถาปัตยกรรมที่ "เข้าใจธรรมชาติของภาพ": CNN
