LLM ทำงานอย่างไร 🏁

ยอดบันได: ประกอบทุกชิ้นจาก 10 สัปดาห์ เป็น ChatGPT/Claude ตัวจริง

🎯 เป้าหมายสัปดาห์นี้ — เรียนจบแล้วนักศึกษาต้อง:
  1. เล่า "วงจรชีวิตของ LLM" ได้ครบ: tokenize → pretrain → fine-tune → RLHF → ใช้งาน
  2. อธิบายได้ว่า hallucination มาจากไหน — จากกลไกที่เรียนมาเอง ไม่ใช่ "ความผิดพลาดแปลกๆ"
  3. อธิบายผลของ temperature ต่อการสุ่มคำ จากการทดลองจริง
  4. ไล่ลูป autoregressive ครบวงจรได้: tokenize → embedding → Transformer → logits → softmax → สุ่ม → ต่อท้าย แล้ววนใหม่
  5. อธิบายได้ว่า RLHF เปลี่ยน "เครื่องต่อข้อความ" เป็น "ผู้ช่วย" ได้อย่างไร — จากการลองเป็นผู้ให้คะแนนเอง

ส่วนที่ 1 — แนวคิด แนวคิด

คำถามเปิดคาบ (คำถามเดียวกับคาบแรกของเทอม!): "ChatGPT ฉลาดเพราะอะไร?" — วันนี้ตอบได้ครบทุกชั้นแล้ว

วงจรชีวิตของ LLM: 4 ขั้น

  • 1. Tokenization — หั่นข้อความเป็นชิ้นเล็ก (token ≈ คำหรือชิ้นส่วนคำ ราว 50,000–100,000 แบบ) เพราะคำเต็มมีไม่จำกัดแต่ softmax ต้องการตัวเลือกจำกัด (เรื่อง Neural Network) ผลข้างเคียง: LLM "มองไม่เห็นตัวอักษร" — นี่คือสาเหตุที่มันนับตัวอักษรผิด สะกดกลับหลังไม่เก่ง
  • 2. Pretraining — Transformer ขนาดยักษ์ (เรื่อง Attention & Transformer) เล่นเกมเดียว: ทำนาย token ถัดไป (เรื่อง AI คืออะไร!) บนข้อความระดับล้านล้าน token เทรนด้วย backprop (เรื่อง Backpropagation) บน GPU หลายหมื่นตัว หลายเดือน — ได้ "base model" ที่รู้ภาษา รู้โลก แต่ยังไม่ใช่ผู้ช่วย (ถามคำถามมันอาจตอบเป็นคำถามต่อ เพราะในอินเทอร์เน็ตคำถามมักตามด้วยคำถาม)
  • 3. Fine-tuning (SFT) — เทรนต่อด้วยตัวอย่าง "บทสนทนาผู้ช่วยที่ดี" หลักหมื่น-แสนคู่ ที่มนุษย์เขียน → โมเดลเรียน "บทบาท" ของผู้ช่วย
  • 4. RLHF — ให้มนุษย์เปรียบเทียบคำตอบหลายแบบ ว่าแบบไหนดีกว่า แล้วเทรนโมเดลให้ผลิตคำตอบแบบที่มนุษย์ชอบ (สุภาพ ปลอดภัย ตรงคำถาม) — นี่คือขั้นที่เปลี่ยน "เครื่องทำนายข้อความ" เป็น "ผู้ช่วย" อย่าง ChatGPT

ทำไมมัน hallucinate — คำตอบจากกลไกที่เราเรียนมาเอง

LLM ไม่มีฐานข้อมูลข้อเท็จจริง มันมีแต่ weight ที่ถูกปรับให้ "ทำนาย token ถัดไปให้เนียน" เมื่อถามสิ่งที่ไม่อยู่ในข้อมูลฝึก มันไม่มีปุ่ม "ไม่รู้" โดยธรรมชาติ — softmax ยังคงผลิตความน่าจะเป็นของคำถัดไปเสมอ และคำที่ "ฟังดูเข้ากับบริบท" ก็ได้คะแนนสูง ผลคือคำตอบที่ไวยากรณ์สมบูรณ์ น้ำเสียงมั่นใจ แต่ข้อเท็จจริงมโน (เหมือน n-gram เรื่อง Language Model & RNN ที่แต่งประโยคลื่นแต่ไม่รู้เรื่องจริง — แค่เนียนกว่ามหาศาล)

Temperature: ปุ่มคุมความ "กล้าเสี่ยง" ของการสุ่ม

ก่อน softmax เราหารคะแนนดิบด้วยค่า T — T ต่ำ: ความน่าจะเป็นกระจุกที่คำอันดับหนึ่ง (ตอบซ้ำๆ เชื่อถือได้) T สูง: กระจายไปคำแปลกๆ (สร้างสรรค์แต่เพี้ยนง่าย) ลองเองใน demo ข้างล่าง

👣 เข้าใจทีละขั้น — ตามติดชีวิตของ 1 token ใน 6 ขั้น

คุณพิมพ์ "แมวชอบกิน" แล้วกด Enter — นี่คือสิ่งที่เกิดขึ้นทุกขั้น ก่อนตัวอักษรถัดไปจะโผล่บนจอ (ทุกขั้นคือบทเรียนเก่าทั้งหมด)

  1. หั่นเป็น token "แมวชอบกิน" → tokenizer หั่นเป็นชิ้น เช่น [แมว][ชอบ][กิน] แต่ละชิ้นแทนด้วยหมายเลข เช่น 8741, 2205, 9038 — โมเดลไม่เคยเห็น "ตัวอักษร" เห็นแต่หมายเลขพวกนี้ (ลองหั่นจริงที่ demo Tiktokenizer ข้างล่าง)
  2. หมายเลข → เวกเตอร์ความหมาย แต่ละ token เปิดตาราง embedding ได้เวกเตอร์ยาวหลายพันมิติ — "แมว" กลายเป็นจุดหนึ่งในแผนที่ความหมาย ที่อยู่ใกล้ "หมา" แต่ไกลจาก "รัฐธรรมนูญ" (แนวคิดจากเรื่อง Language Model & RNN)
  3. ผ่าน Transformer หลายสิบชั้น แต่ละชั้น: attention — token ทุกตัวหันมองกันแล้วดูดบริบท (เรื่อง Attention & Transformer) → MLP — แต่ละ token ครุ่นคิดกับตัวเอง (เรื่อง Neural Network)
    ผ่านไปทีละชั้น เวกเตอร์ของ token สุดท้าย ("กิน") ค่อยๆ เปลี่ยนจาก "ความหมายของคำว่ากิน" เป็น "ความหมายของทั้งประโยค ณ จุดนี้"
  4. คะแนนดิบ 50,000 ค่า → softmax เวกเตอร์สุดท้ายถูกแปลงเป็นคะแนนดิบของทุก token ในคลังศัพท์: "ปลา" 8.2, "ข้าว" 7.9, "นอน" 3.1, ...
    หารด้วย temperature แล้วเข้า softmax (เรื่อง Neural Network) → ความน่าจะเป็น: ปลา 54%, ข้าว 40%, นอน 2%, ...
  5. สุ่มเลือก 1 คำ ทอยลูกเต๋าถ่วงน้ำหนัก — สมมุติได้ "ปลา" (รอบหน้าอาจได้ "ข้าว" — นี่คือเหตุที่ถามซ้ำแล้วคำตอบไม่เหมือนเดิม)
    เชื่อมเอง: ถ้าตั้ง temperature ต่ำมากๆ ขั้นนี้จะเปลี่ยนไปอย่างไร แล้วคำตอบของ AI จะมีนิสัยแบบไหน? (ลองใน demo temperature ข้างล่าง)
  6. ต่อท้าย แล้ววนใหม่ตั้งแต่ขั้น 1 ตอนนี้ข้อความคือ "แมวชอบกินปลา" → ทั้งกระบวนการรันใหม่หมดเพื่อหา token ถัดไป → วนจนจบประโยค
    คำตอบยาว 500 token = วนลูปนี้ 500 รอบ — ไม่มีการ "คิดทั้งย่อหน้าแล้วค่อยพิมพ์" มีแต่ทำนายทีละชิ้นเหมือน bigram สัปดาห์แรกของเรา เพียงแต่ฉลาดกว่าล้านเท่า
    เล่าวงจร 6 ขั้นนี้ให้เพื่อนฟังโดยไม่ดูโน้ต — ถ้าทำได้ คุณคือคนส่วนน้อยมากๆ ที่อธิบาย ChatGPT ได้จริง

🧩 ตัวอย่างหลายแบบ — พฤติกรรมแปลกๆ ของ LLM อธิบายได้หมด

ตัวอย่าง 1 — temperature สุดขั้ว: T = 0.1

คะแนนดิบ 3 คำ: 2, 1, 0 — หารด้วย T = 0.1 ก่อนเข้า softmax (กลายเป็น 20, 10, 0) เกิดอะไรขึ้นกับความน่าจะเป็น?

ขั้น 1 — หารคะแนนด้วย T: (2, 1, 0) ÷ 0.1 = (20, 10, 0) — ความต่างถูกขยาย 10 เท่า
ขั้น 2 — ยกกำลัง e: e²⁰ ≈ 485 ล้าน | e¹⁰ ≈ 22,026 | e⁰ = 1 — ตัวแรกทิ้งห่างหลายหมื่นเท่า
ขั้น 3 — หารเป็นความน่าจะเป็น: คำแรก ≈ 99.995% ที่เหลือแทบเป็นศูนย์
✅ คำตอบ: แทบจะเลือกคำเดิมทุกครั้ง = โหมด "ตอบให้เหมือนเดิมที่สุด" สำหรับงานต้องการความแน่นอน (สรุปเอกสารทางการ) ส่วน T สูงคือโหมดระดมไอเดีย — เลขตัวเดียวเปลี่ยนนิสัยทั้งระบบ
ตัวอย่าง 2 — ทำไมใช้ AI ภาษาไทยแล้ว "เปลือง" กว่าภาษาอังกฤษ

tokenizer ส่วนใหญ่ฝึกกับข้อความอังกฤษเป็นหลัก: "hello" = 1 token แต่ "สวัสดี" อาจถูกหั่นเป็น 3–6 token — ถ้าค่าใช้บริการคิดต่อ token ผลที่ตามมาคืออะไรบ้าง (ตอบ 3 ข้อ)

ขั้น 1 — ตั้งต้นจากข้อเท็จจริง: ความหมายเท่ากัน ไทยถูกหั่นเป็น token มากกว่าอังกฤษ 3–5 เท่า
ขั้น 2 — ไล่ผลที่ 1 (เงิน): คิดเงินต่อ token → ข้อความไทยจ่ายแพงกว่า 3–5 เท่าโดยอัตโนมัติ
ขั้น 3 — ไล่ผลที่ 2 (เวลา): โมเดลผลิตทีละ token → คำตอบไทยใช้รอบวนมากกว่า = ช้ากว่า
ขั้น 4 — ไล่ผลที่ 3 (ความจุ): context window นับเป็น token → ใส่เนื้อหาไทยได้น้อยกว่า บริบทเต็มเร็วกว่า
✅ คำตอบ: แพงกว่า ช้ากว่า จุน้อยกว่า — ทั้งสามผลสืบจากขั้น tokenize ขั้นเดียว ใครเข้าใจวงจรชีวิต LLM จะเดาผลกระทบแบบนี้ได้เอง
ตัวอย่าง 3 — ผ่าตัด hallucination หนึ่งเคส

ถามโมเดล: "เล่าประวัติ ดร.สมชาย ใจดี นักฟิสิกส์รางวัลโนเบลชาวไทย" (บุคคลนี้ไม่มีจริง) — มันตอบมาเป็นชีวประวัติสวยหรู จงไล่กลไกว่าคำตอบปลอมนี้เกิดขึ้นได้ยังไง

ขั้น 1 — เช็คสิ่งที่โมเดลมี: ไม่มีฐานข้อมูลให้ค้นว่า "ไม่มีคนนี้" — มีแต่ weight ที่ทำนายคำถัดไปให้เนียน
ขั้น 2 — ดูบริบทที่เราป้อน: "นักฟิสิกส์...รางวัลโนเบล..." ดึงโมเดลเข้า "ย่านชีวประวัตินักวิทยาศาสตร์" ในข้อมูลฝึก
ขั้น 3 — softmax ทำงานตามหน้าที่: ต้องผลิตคำถัดไปเสมอ ไม่มีปุ่ม "ไม่รู้" → คำที่เนียนกับย่านนั้น ("เกิดปี...", "จบจาก...") ชนะ
ขั้น 4 — ผลรวม: เรื่องแต่งที่ไวยากรณ์สมบูรณ์ น้ำเสียงมั่นใจ ข้อเท็จจริงมโนล้วน
✅ คำตอบ: hallucination ไม่ใช่ "บั๊กแปลกๆ" แต่คือระบบทำงานตามออกแบบกับคำถามที่ไม่มีคำตอบในตัวมัน — ทางกัน: RAG (สัปดาห์หน้า), prompt ให้สิทธิ์ตอบว่าไม่ทราบ และนิสัยผู้ใช้: ชื่อเฉพาะ/ตัวเลข ตรวจเสมอ
ตัวอย่าง 4 — context window ใหญ่แค่ไหนในหน่วย "หนังสือ"

โมเดลรับบริบทได้ 128,000 token — ภาษาอังกฤษ ~1.3 token/คำ, หนังสือเล่มหนึ่ง ~80,000 คำ — ใส่หนังสือได้กี่เล่ม? แล้วภาษาไทยล่ะ?

ขั้น 1 — แปลง token เป็นคำ (อังกฤษ): 128,000 ÷ 1.3 ≈ 98,000 คำ
ขั้น 2 — แปลงคำเป็นหนังสือ: 98,000 ÷ 80,000 ≈ 1 เล่มเศษ
ขั้น 3 — กรณีภาษาไทย: ถูกหั่นถี่กว่า ~3 เท่า → ความจุจริงเหลือ 1/3 → ราวครึ่งเล่มเล็กๆ
✅ คำตอบ: อังกฤษ ~1 เล่ม ไทย ~ครึ่งเล่ม — ก่อนป้อนเอกสารยาวให้ AI ลองกะหน่วยแบบนี้ จะรู้ทันทีว่าทำไมมัน "ลืม" ต้นเอกสาร
ตัวอย่าง 5 — คิดค่าใช้จ่ายต่อคำถาม

ราคาสมมุติ: input 0.5 บาท/พัน token, output 1.5 บาท/พัน token — คำถามแนบเอกสาร 1,000 token, คำตอบ 500 token — ครั้งละกี่บาท? แล้วถ้าผู้ใช้ล้านคน?

ขั้น 1 — ค่า input: 1,000 token × 0.5 บาท/พัน = 0.50 บาท
ขั้น 2 — ค่า output: 500 token × 1.5 บาท/พัน = 0.75 บาท
ขั้น 3 — รวมแล้วคูณสเกล: 1.25 บาท/ครั้ง × 1,000,000 คน = 1.25 ล้านบาท/วัน
✅ คำตอบ: 1.25 บาทดูจิ๊บจ๊อย จนคูณผู้ใช้ — สังเกตว่า output แพงกว่า input 3 เท่า (ต้องวนลูปผลิตทีละ token) เหตุผลที่บริการ AI จำกัดความยาวคำตอบ และทำไมทักษะ "ถามให้กระชับตรงจุด" มีมูลค่าเป็นเงินจริง
ตัวอย่าง 6 — base model vs chat model: ถามคำถามเดียวกัน

พิมพ์ "เมืองหลวงของไทยคือ" ใส่ base model (pretrain ล้วน) กับ chat model (ผ่าน SFT แล้ว) — ผลต่างกันยังไง?

ขั้น 1 — ทวนสิ่งเดียวที่ base model ทำเป็น: "ต่อข้อความให้เนียนตามสถิติ" — ไม่มีแนวคิดเรื่อง "คำสั่ง" หรือ "ผู้ใช้"
ขั้น 2 — ป้อน "เมืองหลวงของไทยคือ": การต่อที่เนียนสุดคือ "กรุงเทพมหานคร ซึ่งมีประชากร..." — ผ่าน! (โจทย์บังเอิญเป็นรูปการต่อประโยคพอดี)
ขั้น 3 — ป้อน "ช่วยสรุปบทความนี้หน่อยครับ": ในเน็ต คำขอมักตามด้วยคำขออื่น (กระทู้/คอมเมนต์) → มันอาจต่อว่า "และช่วยแปลเป็นอังกฤษด้วยนะครับ" — ไม่สรุปให้!
✅ คำตอบ: chat model เข้าใจว่านี่คือคำสั่งถึงตัวเองเพราะผ่าน SFT หลักหมื่นตัวอย่าง — ความรู้มาจาก pretrain แต่ "ความเป็นผู้ช่วย" มาจาก SFT
ตัวอย่าง 7 — "ในฐานะ AI ฉันไม่มีความรู้สึก" — ใครสอนให้พูด?

ประโยคติดปากของแชทบอททุกตัว — มาจากขั้นไหนของวงจรชีวิต และมันคือ "ความจริง" หรือ "บท"?

ขั้น 1 — ไล่หาขั้นที่สอนประโยคนี้: pretrain? ไม่ — ในเน็ตไม่มีใครพูดแบบนี้บ่อยพอ | SFT/RLHF? ใช่ — ทีมงานเขียนตัวอย่างคำตอบแบบนี้ และผู้ให้คะแนนชอบคำตอบถ่อมตัวปลอดภัย
ขั้น 2 — สรุปกลไก: โมเดลเรียน "บทของผู้ช่วยที่ดี" — ประโยคนี้คือพฤติกรรมที่ถูกเลือกผ่านการเทรน
ขั้น 3 — แยกประเด็น: มันไม่ใช่ผลการ "สำรวจตัวเอง" ของโมเดล — โมเดลไม่มีช่องทางรู้เรื่องสภาวะภายในตัวเองแบบนั้น
✅ คำตอบ: เป็น "บท" จาก SFT/RLHF ไม่ใช่รายงานข้อเท็จจริง — แยก "สิ่งที่โมเดลถูกสอนให้พูด" ออกจาก "สิ่งที่เป็นจริงเชิงเทคนิค" ได้ = ทักษะการอ่าน AI ขั้นสูง
ตัวอย่าง 8 — knowledge cutoff: ถามข่าวเมื่อวาน

ถามโมเดลถึงเหตุการณ์เมื่อวานนี้ — เกิดอะไรขึ้นได้บ้าง (มี 3 แบบ) และแบบไหนอันตรายสุด?

ขั้น 1 — แบบที่ 1: ตอบตรงๆ "ข้อมูลผมตัดที่วันที่ X ไม่ทราบเหตุการณ์นี้" — ซื่อสัตย์ ดีสุด
ขั้น 2 — แบบที่ 2: ระบบมี search/RAG เสริม → ไปค้นเว็บแล้วตอบพร้อมแหล่ง — ดี ถ้าแหล่งน่าเชื่อ
ขั้น 3 — แบบที่ 3: มโนเหตุการณ์ที่ฟังเป็นไปได้ — weight แช่แข็ง ณ วันตัดข้อมูล แต่ softmax ยังผลิตคำตอบเสมอ (กลไกเดียวกับตัวอย่างที่ 3)
✅ คำตอบ: แบบ 3 อันตรายสุด เพราะดูเหมือนแบบ 2 แต่ไม่มีการค้นจริง — นิสัยผู้ใช้: ถามเรื่องหลังวันตัดข้อมูล เช็คก่อนว่าระบบมีโหมดค้นเว็บไหม ค่อยเชื่อ
ตัวอย่าง 9 — โมเดลเล็ก vs ใหญ่: เลือกยังไงให้ฉลาด

โมเดล 7B (พันล้าน parameter) ตอบเร็ว ถูก รันบนเครื่องเองได้ | โมเดล 70B+ ฉลาดกว่าแต่แพงกว่า ~10 เท่า — งานไหนใช้ตัวไหน: (ก) จัดหมวดอีเมลลูกค้า (ข) ร่างสัญญากฎหมาย (ค) แชทบอท FAQ ที่มี RAG

ขั้น 1 — งาน (ก) จัดหมวดอีเมล: งานแคบ แพตเทิร์นชัด พลาดบ้างแก้ได้ → 7B พอ (เร็ว ถูก รันเองได้)
ขั้น 2 — งาน (ข) ร่างสัญญากฎหมาย: ต้องใช้เหตุผลละเอียดหลายชั้น พลาดแล้วแพงมาก → ตัวใหญ่ คุ้มส่วนต่าง
ขั้น 3 — งาน (ค) FAQ + RAG: ความรู้อยู่ในเอกสารแล้ว โมเดลแค่อ่าน-เรียบเรียง → 7B + RAG สบาย
✅ คำตอบ: ก=เล็ก ข=ใหญ่ ค=เล็ก+RAG — หลัก: ความรู้หาจากระบบรอบๆ ได้ (RAG/เครื่องมือ) แต่การใช้เหตุผลซับซ้อนต้องจ่ายด้วยขนาด — วิศวกรเก่งเลือกตัวเล็กสุดที่งานยังผ่าน
ตัวอย่าง 10 — system prompt: ข้อความลับหน้าบทสนทนา

แชทบอททุกตัวมีข้อความที่ผู้ใช้มองไม่เห็นแปะหน้าบทสนทนา เช่น "คุณคือผู้ช่วยสุภาพของบริษัท X ห้ามพูดถึงคู่แข่ง..." — อธิบายด้วยกลไกว่าทำไมแค่ "ข้อความนำ" ถึงคุมพฤติกรรมได้

ขั้น 1 — ทวนกลไกหลัก: ทุก token ที่โมเดลผลิต ทำนายต่อจากบริบททั้งหมด — และ system prompt คือส่วนแรกของบริบทเสมอ
ขั้น 2 — มองผ่าน attention: ทุกคำที่กำลังจะพิมพ์ มี attention มองเห็นคำสั่ง "คุณคือผู้ช่วยสุภาพ..." ตลอดเวลา → การทำนายเอียงตามตั้งแต่คำแรก
ขั้น 3 — หาขีดจำกัดจากกลไกเดียวกัน: มันเป็นแค่ "บริบทที่มีน้ำหนัก" ไม่ใช่กฎเหล็ก — ข้อความอื่นในบทสนทนาที่แรงกว่าอาจดึงการทำนายออกได้ (jailbreak)
✅ คำตอบ: คุมได้เพราะเป็นบริบทถาวรที่ attention เห็นตลอด และหลุดได้ด้วยเหตุผลเดียวกัน — ความปลอดภัยจริงจึงต้องมีหลายชั้น ไม่ใช่ฝากไว้กับ prompt เดียว

ส่วนที่ 2 — ทดลองเล่น ทดลองเล่น

import numpy as np

# "แมวชอบกิน ___" — คะแนนดิบ (logits) ของผู้ท้าชิง 8 คำ ชุดเดียวกับ demo 🌡️ เป๊ะ
words  = ["ปลา", "ข้าว", "ขนม", "นม", "ผัก", "หญ้า", "รองเท้า", "การบ้าน"]
logits = np.array([3.0, 2.4, 1.7, 1.4, 0.8, 0.2, -0.6, -1.2])

def softmax(z):                     # เพื่อนเก่าจากเรื่อง Neural Network
    e = np.exp(z - z.max())
    return e / e.sum()

for T in [0.2, 1.0, 2.0]:           # temperature 3 ระดับ
    p = softmax(logits / T)         # สูตรจริงของ LLM: หารด้วย T "ก่อน" เข้า softmax
    print(f"--- T = {T} ---")
    for w, pi in zip(words, p):
        print(f"{w:>8}  {pi:7.1%}  {'█' * round(pi * 40)}")
    # ทอยลูกเต๋าถ่วงน้ำหนัก 1 ครั้ง = ขั้น "สุ่มเลือก" ของ LLM เป๊ะๆ
    print("สุ่มได้:", np.random.choice(words, p=p), "\n")

# ต้องได้: T=0.2 → ปลา 95.1%, ข้าว 4.7%   (กระจุกที่ตัวเต็ง — โหมดงานเอกสาร)
#          T=1   → ปลา 44.7%, ข้าว 24.5%  (สมดุล)
#          T=2   → ปลา 27.9%, การบ้าน 3.4% (กระจายทั่ว — โหมดระดมไอเดีย/เพี้ยน)

แล้วต่อด้วยของจริง: GPT-2 ตัวจริง (บรรพบุรุษโดยตรงของ ChatGPT — สถาปัตยกรรมเดียวกัน เล็กกว่า ~หมื่นเท่า) แต่งประโยคด้วยปุ่ม temperature / top-k เดียวกับ demo ของเรา — รันใน Colab ฟรีได้สบาย:

# !pip install transformers torch

from transformers import pipeline

gen = pipeline("text-generation", model="gpt2")   # GPT-2 (2019), 124 ล้านพารามิเตอร์

prompt = "The cat chased the"
for T in [0.2, 1.0, 2.0]:
    out = gen(prompt,
              max_new_tokens=25,    # วนลูป autoregressive 25 รอบ = 25 token
              do_sample=True,       # True = ทอยลูกเต๋าตาม softmax / False = greedy (อันดับ 1 เสมอ)
              temperature=T,        # ปุ่มเดียวกับ demo 🌡️ และสไลเดอร์ในหน้านี้
              top_k=50)             # ตัดตัวเลือกเหลือ 50 อันดับแรกก่อนสุ่ม (กันคำหลุดโลก)
    print(f"=== T = {T} ===")
    print(out[0]["generated_text"], "\n")

สิ่งที่ต้องสังเกต:

  • บล็อกแรก: แท่ง █ ที่ print ต้องหน้าตาเหมือนแท่งใน demo 🌡️ ทุกตัวเลข (95.1% / 44.7% / 27.9%) — วิดเจ็ตกับโค้ดคือคณิตศาสตร์เดียวกัน ไม่มีมายากล
  • บล็อกสอง: รันซ้ำหลายครั้งที่ T=0.2 ประโยคแทบไม่เปลี่ยน แต่ T=2.0 เพี้ยนคนละเรื่องทุกครั้ง — นิสัยเดียวกับ n-gram demo ข้างบนเป๊ะ
  • GPT-2 ตัวนี้ยังไม่ผ่าน SFT/RLHF — ลองสั่งงานมัน ("Please summarize...") จะเห็นว่ามันแค่ "ต่อข้อความ" ไม่ได้ "คุย" — หลักฐานสดว่าความเป็นผู้ช่วยมาจากสองขั้นท้ายของสายการผลิต

ส่วนที่ 4 — แบบฝึกหัด แบบฝึกหัด

📝 ข้อ 1 (ทำในคาบ) — ประกอบร่าง LLM

จับคู่แต่ละขั้นของวงจรชีวิต LLM (tokenize / pretrain / SFT / RLHF) กับแนวคิดจากสัปดาห์ก่อนๆ ที่มันใช้ — อย่างน้อยขั้นละ 1 แนวคิด พร้อมเลขสัปดาห์

แนวคำตอบ tokenize → softmax ต้องการตัวเลือกจำกัด (ส.3) | pretrain → ทำนายตัวถัดไป (ส.1) + Transformer (ส.10) + backprop (ส.5) + ข้อมูลมหาศาลกัน overfit (ส.6) | SFT → fine-tune คือเทรนต่อด้วย loss เดิมบนข้อมูลใหม่ (ส.2, 5) | RLHF → loss เปลี่ยนจาก "ทำนายแม่น" เป็น "มนุษย์ชอบ" — ยังคงปรับ weight ด้วย gradient (ส.5)
📝 ข้อ 2 (ทำในคาบ) — วินิจฉัยอาการของ LLM

อาการต่อไปนี้เกิดจากขั้นไหน/กลไกไหน: (ก) นับตัว r ในคำว่า strawberry ผิด   (ข) แต่งชื่องานวิจัยปลอมพร้อมชื่อผู้แต่งเนียนๆ   (ค) ปฏิเสธไม่ตอบวิธีทำระเบิด   (ง) ความรู้หยุดอยู่ที่ปีใดปีหนึ่ง

ดูเฉลย (ก) tokenization — มันเห็นเป็น token ไม่เห็นตัวอักษร   (ข) hallucination — softmax ผลิตสิ่งที่ "ฟังดูเข้ากับบริบท" เสมอ   (ค) RLHF/safety fine-tuning   (ง) pretraining มี cutoff — ความรู้คือ weight ที่แช่แข็ง ณ วันเทรนจบ
🏠 ข้อ 3 (การบ้าน) — ทดลอง temperature กับ LLM จริง

ใช้ playground ของ LLM ใดก็ได้ที่ปรับ temperature ได้ (หรือใช้ demo ด้านบนแทน) ถามคำถามเดิม 3 ครั้งที่ T ต่ำ และ 3 ครั้งที่ T สูง บันทึกผล แล้วสรุป: งานแบบไหนควรใช้ T เท่าไหร่ เพราะอะไร (อ้างกลไก softmax ประกอบ)

🏠 ข้อ 4 (การบ้าน) — จับ LLM โกหกแล้วอธิบายให้ได้ว่าทำไม

หาคำถามที่ทำให้ LLM ตอบผิดหรือมโน 1 กรณี (เช่น ถามรายละเอียดบุคคล/สถานที่ที่ไม่ดัง) แคปหลักฐาน + เขียนครึ่งหน้า: อธิบายว่า "ทำไมมันตอบแบบนั้น" โดยใช้กลไกที่เรียนมา (ห้ามตอบว่า "เพราะ AI โง่" — ต้องชี้กลไก)

ส่วนที่ 5 — ขั้นบันไดของสัปดาห์นี้ ขั้นบันได

🏁 ถึงยอดบันไดแล้ว!
คำถามแรกของเทอม "ChatGPT ฉลาดเพราะอะไร" — ตอนนี้ตอบได้เป็นเรื่องราวเดียว: เครื่องทำนายตัวถัดไป (ส.1) ที่มี weight แสนล้านตัว (ส.2) เลือกคำด้วย softmax (ส.3) สร้างจาก MLP (ส.4) เทรนด้วย backprop (ส.5) บนข้อมูลมากพอไม่ให้จำข้อสอบ (ส.6) ด้วยสถาปัตยกรรมที่เข้ากับภาษา (ส.7, 9) คือ Transformer (ส.10) แล้วขัดเกลาให้เป็นผู้ช่วยด้วย RLHF (วันนี้)
